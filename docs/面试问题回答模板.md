## 📋 基础问题

### Q1: 你对缺失值的处理策略是什么？为什么这样选择？

**回答模板：**
> "我在处理缺失值时，根据字段的业务重要性采用不同的策略：
>
> **1. 关键业务字段：** CustomerID（客户ID）缺失时，我填充为'Unknown'而不是删除，因为：
> - 这些订单仍然有销售价值，删除会损失销售数据
> - 填充后可以单独分析'匿名订单'群体，发现特殊消费模式
> - 这在电商平台很常见，如游客下单
>
> **2. 描述字段：** Description（商品描述）缺失时，填充为'unknown'，因为：
> - 不影响销售金额等核心指标
> - 可以通过StockCode（商品编码）识别商品
>
> **3. 其他字段：** 如果有更多字段，我会根据业务场景决定是删除还是填充，确保数据清洗不会过度损失有效信息。"

### Q2: 为什么要删除数量<=0和单价<=0的记录？

**回答模板：**
> "删除数量<=0和单价<=0的记录是基于我的分析目标决定的：
>
> **当前分析目标：** 聚焦于**销售数据分析**，不包括退货分析
>
> **数量<=0的情况：**
> - 通常表示退货、取消订单或库存调整
> - 如果分析退货率，应该保留并单独处理
> - 但当前项目专注于销售趋势分析，所以移除
>
> **单价<=0的情况：**
> - 可能是促销赠品、测试订单或数据录入错误
> - 会扭曲销售金额和平均客单价的计算
> - 影响定价策略和营收分析
>
> **改进建议：** 如果项目需要分析退货率，我会：
> - 创建IsReturn字段标识退货订单
> - 分别分析销售和退货数据
> - 计算净销售额 = 销售额 - 退货额"

### Q3: 数据类型转换的原因是什么？

**回答模板：**
> "我进行了两个关键的数据类型转换：
>
> **1. InvoiceDate: str → datetime**
> - **原因：** 时间序列分析必须使用datetime类型
> - **优势：** 可以提取年、月、日、小时等时间维度
> - **应用：** 分析销售趋势、季节性、高峰时段
>
> **2. CustomerID: float → str**
> - **原因：** 客户ID是标识符，不是数值，不应该进行数学运算
> - **优势：** 避免被错误地进行加减乘除运算
> - **一致性：** 与InvoiceNo、StockCode等其他ID字段保持一致
>
> **最佳实践：** 所有标识符字段都应转换为字符串类型，这是数据清洗的行业标准做法。"

### Q4: 重复数据的处理逻辑是什么？

**回答模板：**
> "我使用 `keep=False` 来检测所有重复数据，然后用 `keep='first'` 删除重复项：
>
> **检测阶段（keep=False）：**
> - 标记所有重复的记录，包括第一次出现的
> - 目的是全面了解重复情况，避免遗漏
>
> **删除阶段（keep='first'）：**
> - 保留第一次出现的记录，删除后续重复项
> - 原因：第一次出现的数据通常是原始记录，后续可能是系统重复导入
>
> **业务考虑：**
> - 如果需要分析重复订单的原因，会先提取重复数据样本
> - 保留第一次出现确保不丢失原始交易数据
> - 这个策略在大多数电商数据处理场景中都是合理的"

### Q6: 异常值的检测方法有哪些？你如何处理？

**回答模板：**
> "我采用了两种异常值检测方法：
>
> **方法1：业务规则检测（基础）**
> - 数量 <= 0：可能是退货，根据分析目标决定保留或删除
> - 单价 <= 0：异常定价，通常删除
> - 数量 > 10000：可能是批量采购，需要业务确认
>
> **方法2：统计学检测（进阶）**
> - 使用 **IQR（四分位距）方法**：
>   - 计算 Q1（25%分位数）和 Q3（75%分位数）
>   - IQR = Q3 - Q1
>   - 异常值范围：< Q1 - 3*IQR 或 > Q3 + 3*IQR
> - 使用 3×IQR 而不是 1.5×IQR，减少误删正常的大额订单
>
> **处理策略：**
> - 统计异常值数量但不立即删除
> - 结合业务场景判断（如大客户采购可能 legitimately high）
> - 记录异常值供后续分析使用
>
> **扩展方法：** 如果数据更复杂，还可以使用 Z-score、箱线图、聚类等方法。"

### Q11: 为什么使用 `gbk` 编码而不是 `latin1`？

**回答模板：**
> "编码选择取决于数据的实际编码格式：
>
> **1. 原始编码检测**
> - 我先用 `ce_encoding.py` 检测了文件的实际编码
> - 确认文件是 GBK 编码（中文编码）
>
> **2. 编码对比**
> - **GBK：** 支持中文字符，适合中文数据
> - **Latin1：** 仅支持西欧字符，无法正确显示中文
>
> **3. 错误示例**
> - 使用 latin1 读取中文数据会导致乱码
> - 错误示例：'客户ID' 会变成乱码 '¿¿¿¿'
>
> **4. 最佳实践**
> - 在读取 CSV 前先检测文件编码
> - 使用正确的编码可以避免数据损坏
> - 如果不确定，可以尝试常见编码（utf-8, gbk, latin1）
>
> **扩展：** 对于未知编码，可以使用 `chardet` 库自动检测：
> ```python
> import chardet
> with open('file.csv', 'rb') as f:
>     result = chardet.detect(f.read())

### Q12: `df.isnull()` 的使用是否正确？

**回答模板：**
> "在原始代码中，`df[df.isnull()]` 的使用是错误的：
>
> **错误分析：**
> ```python
> isnull = df[df.isnull()]  # ❌ 错误用法
> ```
> - `df.isnull()` 返回整个 DataFrame 的布尔矩阵
> - 不能直接用 `df[...]` 筛选，因为维度不匹配
> - 这行代码不会报错，但 `isnull` 会是空 DataFrame，没有任何作用
>
> **正确做法：**
> ```python
> # 方法1：查看有缺失值的行
> missing_rows = df[df.isnull().any(axis=1)]
>
> # 方法2：查看每列的缺失值数量
> missing_count = df.isnull().sum()
>
> # 方法3：查看总缺失值数量
> total_missing = df.isnull().sum().sum()
> ```
>
> **我的改进：**
> - 使用 `df.isnull().sum()` 统计每列缺失值
> - 使用 `df.isnull().sum().sum()` 统计总缺失值
> - 计算缺失比例，制定填充策略
>
> **教训：** 在使用 pandas 时，要特别注意返回值的类型和维度。"

### Q13: 为什么不使用 `inplace=True`？

**回答模板：**
> "我选择不使用 `inplace=True`，主要有以下考虑：
>
> **1. 可读性和调试**
> ```python
> # 使用链式操作，代码更清晰
> df = df[df['Quantity'] > 0].reset_index(drop=True)
>
> # 使用 inplace=True，不够直观
> df.drop_duplicates(inplace=True)
> ```
>
> **2. 避免意外修改**
> - `inplace=True` 直接修改原 DataFrame，可能影响其他引用
> - 显式赋值让数据流向更清晰
>
> **3. 链式操作支持**
> - 不使用 inplace 可以链式调用多个方法
> ```python
> df = (df.drop_duplicates()
>           .reset_index(drop=True)
>           .assign(TotalAmount=lambda x: x['Quantity'] * x['UnitPrice']))
> ```
>
> **4. 性能考虑**
> - `inplace=True` 并不一定节省内存
> - pandas 内部很多操作都会创建副本
>
> **5. 最佳实践**
> - 现代推荐使用 `df = df.method()` 而不是 `df.method(inplace=True)`
> - 让代码更符合函数式编程的理念
>
> **我的选择：** 为了代码的清晰性和可维护性，我使用显式赋值而不是 inplace。"
